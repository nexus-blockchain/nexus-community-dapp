# nexus-community-dapp 设计开发文档

## 1. 项目定位

**项目名：** `nexus-community-dapp`  
**目标：** 基于 Nexus 链上社区、会员、商城、返佣、忠诚度与 NEX 市场能力，设计一款 **移动优先** 的社区电商金融化 DApp。

### 1.1 核心产品关键词
- **社区感**：邀请关系、会员成长、社群身份、等级荣誉、团队可视化
- **金融感**：资产面板、收益流水、奖池领取、积分/购物金/Token 余额、NEX 市场
- **交易感**：商品浏览、下单支付、履约售后、订单托管、交易状态清晰
- **信任感**：链上记录、可追溯、可验证、状态透明、风控提示明确

### 1.2 目标用户
1. **普通用户 / 会员**：加入社区、购买商品、赚取积分/购物金/返佣
2. **团长 / 推广者**：查看推荐链路、多级收益、奖池奖励、团队成长
3. **店主 / 店铺管理员**：管理店铺、商品、订单、会员、返佣策略
4. **平台运营 / 风控**：市场异常处理、争议处理、封禁、配置开关

## 2. 基于 Nexus 项目的移动前端技术栈选择

## 2.1 结论

### 推荐主栈
**Next.js 14 + React 18 + TypeScript + Tailwind CSS + Radix UI + TanStack Query + Zustand + @polkadot/api + PWA（移动优先）**

### 原因
Nexus 现有前端项目 `nexus-entity-dapp` 已经采用：
- `Next.js 14`
- `React 18`
- `TypeScript`
- `Tailwind CSS`
- `Radix UI`
- `TanStack Query`
- `Zustand`
- `@polkadot/api`
- `next-intl`

这意味着 `nexus-community-dapp` 继续使用同体系技术栈，能够：
- **最大化复用现有链连接、查询、交易、钱包、国际化能力**
- **降低团队学习和维护成本**
- **共享组件风格与工程规范**
- **更快上线移动端 H5 / PWA 版本**

## 2.2 不建议首版直接选择的方案

### 不建议首版用 React Native / Flutter
原因：
- 现有 Nexus 前端代码与组件体系主要是 **Web React 体系**，复用率低
- `@polkadot/api` 在 Web 侧更成熟，移动原生环境接入成本更高
- Radix / Tailwind / Next App Router / 现有 hooks 基本无法直接复用
- 首版目标应优先验证业务闭环，而不是重建一套客户端技术体系

## 2.3 推荐落地方式

### Phase 1：移动 Web / PWA
- 适合快速上线
- 适合邀请裂变、社群传播、浏览器直接打开
- 保持与现有 `nexus-entity-dapp` 的工程一致性

### Phase 2：原生壳封装
建议在 PWA 稳定后，再做：
- **Capacitor 封装 iOS / Android**，补齐推送、深链、相册、扫码、分享、Secure Storage 等能力

> 结论：**首版最佳方案不是重做原生，而是“Next.js 移动优先 + PWA”，二期再加 Capacitor 原生壳。**

## 3. 模块能力拆解与前端映射

以下设计基于这 9 个模块：
- `pallet-commission-single-line`
- `pallet-commission-pool-reward`
- `pallet-commission-multi-level`
- `pallet-entity-shop`
- `pallet-entity-member`
- `pallet-entity-loyalty`
- `pallet-entity-product`
- `pallet-entity-order`
- `pallet-nex-market`

## 3.1 模块到页面的映射

| 模块 | 核心能力 | 对应页面/功能 |
|---|---|---|
| `pallet-entity-member` | 会员注册、推荐关系、等级、审批、封禁、升级规则 | 加入社区、会员中心、邀请关系、团队树、会员审批 |
| `pallet-commission-single-line` | 单线收益、上下线收益、等级覆盖、预览 | 单线收益页、收益预览、策略配置 |
| `pallet-commission-multi-level` | 多级返佣、层级配置、激活条件、激活进度 | 多级收益页、团队贡献页、策略配置页 |
| `pallet-commission-pool-reward` | 奖池轮次、快照、领取、配置、暂停 | 奖池页、领取页、轮次详情、运营配置 |
| `pallet-entity-shop` | 店铺创建、更新、暂停、转移、资金、管理者 | 店铺主页、店铺资料、店铺管理台 |
| `pallet-entity-product` | 商品创建、编辑、上下架、删除、库存 | 商品列表、商品详情、商品管理 |
| `pallet-entity-loyalty` | 积分、NEX购物金、Token购物金、积分兑换 | 积分钱包、购物金钱包、抵扣结算、积分流水 |
| `pallet-entity-order` | 下单、支付、发货、收货、退款、服务单 | 订单确认、订单列表、订单详情、售后页 |
| `pallet-nex-market` | NEX/USDT 挂单、吃单、确认付款、争议、深度 | OTC市场、挂单页、交易页、订单簿、争议页 |

## 3.2 建议前端读取方式

| 模块 | 推荐读取方式 |
|---|---|
| `member` | 优先使用 Runtime API（会员仪表盘、推荐树、分页查询） |
| `nex-market` | 优先使用 Runtime API（订单簿、深度、市场摘要、用户交易） |
| `shop` / `product` / `order` / `loyalty` | 先走链上 storage + 事件订阅；二期增加 indexer/BFF 提升首页聚合与搜索体验 |
| `commission-*` | storage + 查询函数 + 交易回执；收益总览建议二期走 indexer 聚合 |

> 对移动端来说，**首页推荐流、排行榜、收益总览、订单聚合** 都更适合通过 **indexer / BFF** 提供，而不是直接在手机端遍历链上存储。

## 7. 前端功能模块设计

## 7.1 钱包与签名层

建议封装统一接口：`WalletBridge`

### 适配器
- `WebExtensionSigner`：桌面浏览器插件钱包
- `DeepLinkSigner`：移动钱包深链唤起
- `EmbeddedSigner`：二期原生壳内安全签名器

### 首版建议
- 先支持 **只读浏览 + 登录后交易**
- Web 场景优先兼容现有 `@polkadot/extension-dapp`
- 移动场景通过深链/原生桥逐步补齐签名

> 不建议在首版用“明文/文件钱包”方案直接上生产；移动端必须结合系统 Secure Storage / Biometric。

## 7.2 链接入层

复用 Nexus 现有思路：
- `ApiProvider`
- `WsProvider`
- 多节点探测
- 节点健康检查
- 自动切换优质节点

## 7.3 数据访问分层

```text
UI Page
  -> Feature Hooks
    -> Query Service / Mutation Service
      -> Polkadot API / Runtime API / Indexer BFF
```

### 建议拆层
- `features/*`：按业务域组织
- `shared/ui/*`：通用组件
- `shared/chain/*`：链访问封装
- `shared/wallet/*`：签名和账户适配
- `shared/utils/*`：格式化与校验
- `server/bff/*`：聚合接口（可选）

## 7.4 状态管理
- `TanStack Query`：远程数据缓存、重试、失效控制
- `Zustand`：钱包、UI 状态、筛选条件、浮层状态
- `React Hook Form + Zod`：下单、挂单、配置表单

## 11. 关键体验原则

1. **移动优先，不做后台搬运式页面**
2. **高频功能前置：加入、下单、领取、邀请、查看收益**
3. **资产状态可视化：金额、等级、进度、风险都要明显**
4. **交易状态强提示：支付中、待发货、待确认、争议中要极清晰**
5. **复杂运营轻量化：移动只保留高频操作，复杂配置留桌面**
6. **链上失败提示可理解：不仅显示错误码，还要转成人话**

## 12. 最终技术选型结论

### 最适合 Nexus 的移动前端方案

- **框架**：Next.js 14（App Router）
- **语言**：TypeScript
- **UI**：Tailwind CSS + Radix UI
- **状态**：TanStack Query + Zustand
- **表单**：React Hook Form + Zod
- **图表**：Recharts
- **链交互**：@polkadot/api
- **国际化**：next-intl
- **发布形态**：移动 Web / PWA 优先
- **二期 App 化**：Capacitor 封装 iOS / Android

### 选择理由一句话总结
**因为它与 Nexus 现有前端工程最一致、可复用最多、最快落地、最适合先做移动端 DApp MVP。**
