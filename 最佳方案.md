# Nexus Community dApp 最佳融合方案

## 1. 结论

基于 `设计开发.md` 与 `DEVELOPMENT.md` 的对比，**最佳方案不是二选一，而是：**

> **同一业务内核 + Web/PWA 首发 + Native 能力预埋 + 原生 App 二期增强**

即：

- **V1 主交付**：`Next.js 14 + PWA`，快速完成社区 / 商城 / 收益 / 市场的 MVP 闭环
- **V1 同时保留**：桌面 Web 运营台，承接复杂配置与后台管理
- **V2 增强**：`React Native (Expo)`，重点承接钱包安全、推送、生物识别、原生交易体验
- **长期架构**：共享 `Domain / Chain SDK / Wallet Bridge / Design Tokens / BFF`

---

## 2. 两份方案的核心定位

### A. `设计开发.md`
定位：**快速上线、复用 Nexus 现有前端体系、PWA 优先**

**优势**
- 与现有 `nexus-entity-dapp` 技术栈最一致
- `@polkadot/api` 在 Web 端成熟，接入风险最低
- H5 / 邀请裂变 / 社群传播能力强
- 天然兼容桌面 Web 与轻运营入口
- 首版开发速度快、维护成本低

**劣势**
- 原生推送、生物识别、安全存储能力弱
- 金融图表、手势交互、交易体验不如原生
- 若长期停留在 WebView 壳，会形成体验天花板

### B. `DEVELOPMENT.md`
定位：**真原生移动端体验、钱包安全与高频交易场景优先**

**优势**
- 原生性能更好，适合行情、K 线、手势、动画
- 钱包安全能力更完整（Keychain / Keystore / Biometric）
- 更适合订单通知、佣金到账、本地推送
- 更适合中高频交易与资产管理场景

**劣势**
- 首版研发成本明显更高
- `@polkadot/api` 在 RN 侧适配复杂，存在 polyfill / wasm 风险
- 现有 Web 组件与工程资产难复用
- 即便做 RN，复杂运营后台仍需单独做 Web
- 早期业务验证速度慢

## 3. 两份文档的共同正确方向

这部分建议直接保留：

- React + TypeScript 技术体系
- TanStack Query + Zustand 状态模型
- 9 个 Pallet 的业务映射方式
- 移动优先的信息架构

- 必须建设 `Indexer / BFF`
- 复杂运营与治理配置保留在桌面 Web

## 4. 最佳融合方案

## 4.1 产品发布策略

### Phase 1：PWA MVP（主战场）

使用：

- `Next.js 14`
- `React 18`
- `TypeScript`
- `Tailwind CSS`
- `Radix UI`
- `TanStack Query`
- `Zustand`
- `@polkadot/api`
- `next-intl`

目标：

- 最快跑通 **加入社区 → 浏览商品 → 下单 → 查看收益 → 进入市场**
- 支撑邀请裂变、社群传播、浏览器访问
- 与现有 Nexus Web 技术栈统一

### Phase 1.5：轻量 App 分发（可选）

如果需要尽快上架安卓市场或提供桌面图标入口：

- Android 可考虑 `TWA / Capacitor` 作为**临时分发壳**
- 但**不建议把 Capacitor 作为长期原生方案**

### Phase 2：Expo 原生增强

当出现以下需求时启动：

- 钱包安全升级为内置密钥 + 生物识别
- 高频交易用户显著增加
- 强依赖推送通知 / 深链 / 扫码 / 相册 / 本地安全存储
- 市场模块成为高频核心入口

此时以 `React Native + Expo` 构建原生 App，但只复用共享业务层，不重复造业务轮子。

---

## 4.2 最终推荐的信息架构

建议融合两份文档后，底部导航统一为：

1. **社区**（兼首页）
2. **商城**
3. **市场**
4. **收益**
5. **我的**

这样处理的原因：

- 吸收 `设计开发.md` 的“首页聚合能力”
- 吸收 `DEVELOPMENT.md` 的“市场必须一级入口”
- 避免底部导航超过 5 个一级 Tab

其中：

- **社区 Tab** = 首页 + 社区总览 + 资产卡 + 今日收益 + 热门商品 + 市场摘要 + 社区动态
- **我的 > 运营工作台** 仅保留高频轻操作
- 复杂配置、批量管理、长表单策略配置全部保留在桌面 Web

## 4.3 技术架构（推荐）

### 前端形态

```text
apps/
  web/         # Next.js PWA，主用户端
  admin/       # Next.js 桌面运营台
  mobile/      # Expo App（二期）

packages/
  domain/      # 业务模型、zod schema、use cases
  chain-sdk/   # Polkadot API 封装、Runtime API、事件解析、Tx Builder
  wallet-bridge/ # Extension / WalletConnect / Embedded Signer 抽象
  ui-tokens/   # 颜色、间距、字号、图标语义
  query-keys/  # Query key 与缓存策略

services/
  bff/         # 聚合接口
  indexer/     # 排行榜、订单聚合、收益汇总、市场摘要
```

### 数据访问原则

**链直连负责**
- 钱包签名
- 交易提交
- 用户关键状态读取

**BFF / Indexer 负责**
- 首页推荐流
- 排行榜 / 团队榜
- 收益总览
- 订单聚合
- 搜索与筛选
- 市场摘要 / 深度 / K 线聚合

---

## 4.4 钱包与安全策略

### V1（Web / PWA）

- 只读浏览 + 登录后交易
- 优先支持：
  - 浏览器扩展钱包
  - WalletConnect / DeepLink
- **不建议首版在 Web 中内置助记词钱包**

### V2（Native）

- 增加 Embedded Signer
- 私钥/助记词存储到 Keychain / Keystore
- 生物识别解锁
- 本地通知与深链唤醒

## 5. 最终取舍建议

## 应该保留自 `设计开发.md` 的内容

- Next.js 14 PWA 作为首发主栈
- 与 `nexus-entity-dapp` 的工程一致性
- 桌面运营台保留
- BFF / Indexer 思路
- 按业务域划分 `features/*`

## 应该吸收自 `DEVELOPMENT.md` 的内容

- 市场模块的原生优先思维
- 钱包安全与生物识别设计
- 推送通知 / 本地能力规划
- 性能预算与金融化交互标准
- 原生 App 目录规划与能力清单

## 不建议直接照搬的内容

- **不建议首版直接全量 RN**
- **不建议长期只靠 Next.js + Capacitor 作为最终态**

---

## 6. 推荐里程碑

### P0
- Monorepo 初始化
- `chain-sdk` / `wallet-bridge` / `domain` 抽象
- 设计 Token 与 UI 规范统一
- Indexer / BFF 最小可用版本

### P1
- PWA 首发：社区 / 商城 / 订单 / 收益
- 只读浏览 + 钱包连接 + 基础交易

### P2
- 市场模块上线
- 收益总览、订单聚合、排行榜走 BFF
- 运营台桌面版上线

### P3
- 根据业务数据评估是否进入 Expo 原生阶段
- 仅把高频、高价值模块迁入原生

---

## 7. 一句话结论

> **短期选 Web/PWA，是为了最快落地；中期引入 Expo Native，是为了高频交易与钱包安全；长期用共享业务内核，避免双端重复建设。**
